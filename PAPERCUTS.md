# Papercuts

Small frictions hit while working — retried tool calls, undocumented setup steps, flaky commands, stale caches, misleading errors, non-obvious gotchas. Append an entry as you hit them (see AGENTS.md → "Log papercuts"). One or two sentences each: what you were doing → what got in the way (a guess at cause/fix is a bonus).

- Testing autocomplete suggestions: asserting `html =~ "Park Site"` on the dropdown fails because `Highlight.highlighted_text` wraps the matched term in `<strong>` (`<strong>Park</strong> Site`). Assert on the un-matched remainder of the name (as form_test does) or query by element id.
- Running `mix test apps/kjogvi/test/... apps/kjogvi_web/test/...` from the umbrella root silently ran only the kjogvi_web files ("14 passed", no warning that the kjogvi paths were skipped). Run each app's paths from its own directory (`cd apps/kjogvi && mix test test/...`) to be sure they execute.
- Asserting the LiveView error flash: the element id is `#flash-group-error` (the app layout's `flash_group` id-prefixes its flashes), not the `#flash-error` default from `CoreComponents.flash`.
- Adding `{:struct, Kjogvi.Geo.Location}` as a NimbleOptions type in `Location.Filter`'s schema created a compile-time xref cycle (module attribute evaluation makes it a compile dep; Location depends back on Filter via Query), failing `mix lint`. Used `:any` for the field type instead.
- Menu links (e.g. `private_menu.html.heex`) are plain `href` links, so every nav click is a full document reload; in Firefox this flashed the "Connection interrupted" popup because Firefox aborts the websocket at navigation start (mitigated with `disconnectedTimeout` in app.js). Converting same-live_session links to `navigate={...}` would avoid the reload entirely.
- Prospective change: `Live.My.Locations.Show` and `Live.Admin.Locations.Show` have converged to near-copies (breadcrumbs, header/stats, details card, map, ancestry/children are duplicated verbatim); the differences are all cleanly keyed off `scope.area` (load + children query, paths, lifelist links, eBird details, delete guard). Combine into one `Live.Locations.Show` serving both routes, as stage 7 did with `Live.Locations.Form` — the index LVs stay separate (their divergence is real).
- Prospective change: consider splitting the single `disabled` flag into two distinct states. **Disabled** = admin-controlled, hard-disallowed: never offered anywhere and not usable at all. **Retired** = owner-controlled, soft-disallowed: hidden or deprioritized in pickers but still usable if already referenced (already a location's parent or a checklist's location). Today `exclude_disabled` in `Location.Filter` covers only the hard case; the soft "still-usable-if-present" case has no representation.
- `apps/kjogvi/test/kjogvi/geo/ebird_test.exs` is flaky (~1 run in 8 on the full suite, pre-existing on HEAD): an `insert!` of `:country` fails on the `unique_index(:locations, [:iso_code])`. Many `async: true` tests across files hardcode the same `iso_code: "AD"`/`"CZ"`; the `:country` factory sequences only `slug`, not `iso_code`. Sequencing `iso_code` in the factory (and passing it explicitly only where the test cares) would likely fix it.
- The `disabled` exclusion had to be repeated across eight separate joins in `EbirdLocation.Query` (four link updates, the name-pass candidates, and three ISO-side shape queries), each spelling out `l.location_type == :subdivision1 and is_nil(l.user_id) and not l.disabled` by hand. Easy to add a ninth query and silently forget the guard — a shared "linkable common location" query fragment (or a `Location.Query.common_linkable/1` the joins compose onto) would make the rule one thing rather than eight.
- Exclusive job rows don't record the perform result, so a page mounted *after* a run finishes shows the generic "Success." flash instead of the rich completion message ("Legacy import done.", "eBird preload done: N new checklists.") — only sessions subscribed when the lifecycle `:ok` fired see it. Cheap middle ground: have `Kjogvi.Jobs.Bridge` write the completion message into `meta` on `:stop` and let `Jobs.status/2` fold it into the ok result; a durable "what happened" beyond the prune window belongs in a domain record per the migration plan.

- Running `mix lint.fix` halted the VM with exit status 1 at the dialyzer step on a run where new modules had just been added, but a plain `mix dialyzer` immediately after reported "Total errors: 0 ... passed successfully" and a re-run of `mix lint` was clean. Looks like a stale/first-build PLT hiccup rather than a real finding — re-running clears it.

- Implementing the eBird CSV import: `Checklist.changeset/2` doesn't cast `:ebird_id` or `:import_source`, and `Observation.changeset/2` didn't cast `:import_source` — so an import building attrs maps for those changesets silently drops them (checklist saved with a nil `ebird_id`, breaking re-import dedup) with no error. These are import-provenance fields; worth a comment on the changesets noting they must be set out-of-band (or a dedicated import changeset).

- The eBird CSV import failed with `Oban.PerformError ... {:error, :enoent}` **and no stacktrace** (the tuple is a returned value Oban wraps, not a crash). Root cause: `Kjogvi.Imports.Upload`'s dev `:path` was the **relative** `"tmp/imports"`. The LiveView (in `kjogvi_web`) stored the file resolving that against the umbrella root, but the Oban job (in `kjogvi`) runs with cwd = `apps/ornitho_web`, so it resolved `"tmp/imports"` under *that* app and `File.cp` missed the file. Relative paths are a trap in an umbrella — processes in different apps have different cwds. Fixed by anchoring the config path to the umbrella root with `Path.expand("../tmp/imports", __DIR__)`. General lesson: any on-disk path shared between a web request and a background job must be absolute.
- Also added a LiveView guard so a genuine second upload while an import is in flight (`Oban.insert` returns `conflict?: true`, keyed on `user_id`) is rejected and its just-stored file deleted, rather than silently orphaned — `user_id`-only slot-keying is intentional (one import per user).
- `KjogviWeb.Live.My.Imports.IndexTest`'s "uploading a zip stashes it" test flaked on `File.read!(dest) == csv_zip()`: `:zip.create` bakes the current wall-clock time into each entry header, and the test built the zip twice (upload vs. assertion) — if the two calls straddled a second boundary, one timestamp byte differed (`<z_j` vs `=z_j`). Fixed by generating the zip once and reusing it. Lesson: never compare two independently-created zips/archives for byte-equality; capture one.
